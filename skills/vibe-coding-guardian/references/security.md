# 安全护栏参考

加载条件：
- 🏭 Production 模式下 🔴 高风险任务 → 执行全部章节
- 🚀 Prototype 模式下 🔴 高风险任务 → 只执行"输入验证"和"认证与授权"章节

---

## 核心原则

**零信任假设**：所有外部输入都是恶意的，直到被证明安全。

---

## 输入验证

1. 验证在入口处完成，不信任下游验证
2. 白名单优于黑名单
3. 验证失败返回通用错误，不泄露系统信息
4. 验证逻辑集中管理，不散布在业务代码中

检查：
- 所有外部输入是否经过类型校验？
- 字符串输入是否经过长度限制和格式校验？
- 数值输入是否经过范围校验？
- 文件上传是否限制类型和大小？
- 错误信息是否避免泄露内部细节？

---

## 认证与授权

### 认证（Authentication）— 你是谁

- 密码必须哈希存储（bcrypt / argon2），永不明文
- 敏感操作需要重新认证
- 会话管理：合理的超时、安全的存储

### 授权（Authorization）— 你能做什么

- 最小权限原则：默认拒绝，按需授权
- 认证和授权分离，不混为一谈
- 权限检查在后端入口处执行，不依赖前端隐藏
- 资源级权限：不同用户只能访问自己的资源

---

## 数据保护

| 数据类型 | 处理方式 |
|----------|----------|
| 密码 | 哈希存储（bcrypt/argon2） |
| PII | 加密存储 + 访问控制 |
| API 密钥/令牌 | 环境变量或密钥管理服务 |
| 日志 | 脱敏处理，不记录敏感字段 |

传输安全：HTTPS、安全头部（CSP、HSTS、X-Frame-Options）、不在 URL 传递敏感参数。

## 常见漏洞防护

| 漏洞 | 根因 | 防护 |
|------|------|------|
| SQL 注入 | 拼接用户输入到 SQL | 参数化查询，永不拼接 |
| XSS | 未编码输出用户输入 | 输出编码 + CSP 策略 |
| CSRF | 未验证请求来源 | Token 验证 + SameSite Cookie |
| 路径遍历 | 未校验文件路径 | 路径规范化 + 白名单目录 |
| SSRF | 未限制外部请求目标 | URL 白名单 + 禁止内网访问 |
| 敏感数据泄露 | 错误信息过于详细 | 通用错误 + 内部日志分离 |

---

## 安全审查触发条件

以下场景必须执行安全审查：
- 新增认证/授权逻辑
- 处理支付或金融数据
- 修改数据访问权限
- 引入新的外部服务集成
- 修改加密/哈希相关代码
- 处理用户上传内容

## 安全扫描工具速查

根据项目技术栈选择对应工具，按优先级执行：

### 依赖漏洞扫描

| 技术栈 | 命令 | 说明 |
|--------|------|------|
| Node.js | `npm audit` / `yarn audit` | 检查已知漏洞的依赖版本 |
| Python | `pip-audit` / `safety check` | 检查 Python 包漏洞 |
| Go | `govulncheck ./...` | Go 标准漏洞检查 |
| Java | `mvn org.owasp:dependency-check-maven:check` | OWASP 依赖检查 |

### 静态安全分析（SAST）

| 工具 | 命令 | 适用场景 |
|------|------|----------|
| Semgrep | `semgrep --config=auto` | 多语言通用安全规则 |
| Bandit | `bandit -r src/` | Python 专用 |
| ESLint Security | `eslint --ext .js,.ts src/` | JS/TS 安全规则 |
| CodeQL | GitHub Actions 自动触发 | 深度语义分析 |

### 密钥泄露检测

| 工具 | 命令 | 说明 |
|------|------|------|
| Gitleaks | `gitleaks detect` | 扫描 git 历史中的密钥泄露 |
| TruffleHog | `trufflehog git file://.` | 深度扫描 git 仓库 |

### 容器安全

| 工具 | 命令 | 说明 |
|------|------|------|
| Trivy | `trivy image <image-name>` | 容器镜像漏洞扫描 |
| Hadolint | `hadolint Dockerfile` | Dockerfile 最佳实践检查 |

### 执行顺序

```
触发安全审查
  │
  ├─ ❶ 依赖漏洞扫描（npm audit / pip-audit）
  │     └─ 存在高危漏洞 → 阻断，先修复
  │
  ├─ ❷ 静态安全分析（semgrep --config=auto）
  │     └─ 存在安全问题 → 评估严重性后决定
  │
  ├─ ❸ 密钥泄露检测（gitleaks detect）
  │     └─ 存在泄露 → 阻断，立即轮换
  │
  └─ ❹ 全部通过 → 安全审查通过
```
