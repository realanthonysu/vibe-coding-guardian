# 重构治理参考

加载条件：代码腐化、技术债务、渐进式重构、功能扩展到已有代码

---

## 何时重构

- 添加新功能前，先让代码接纳新功能
- 修复 bug 时，消除导致 bug 的结构问题
- 代码重复三次，必须提取
- 命名不再表达意图时，立即重命名
- 注释需要解释"怎么做"而非"为什么"时，代码需要重构

## 何时 NOT 重构

- 代码虽然丑但稳定且不需要修改
- 重写成本低于重构成本
- 没有测试覆盖的重构（先补测试再重构）
- 截止日期前的紧急交付

## 渐进式重构模式

### 绞杀者（Strangler Fig）

新功能走新路径，逐步替换旧路径，直到旧路径可安全移除。

```
旧系统 ──── 逐步减少流量 ────→ 移除
  │
新系统 ──── 逐步增加流量 ────→ 完全替代
```

适用：大型系统迁移、API 版本升级

### 分支抽象

先提取接口，再切换实现，最后移除旧实现。

```
1. 提取接口 → 旧实现实现该接口
2. 新实现实现该接口 → 通过配置切换
3. 移除旧实现 → 接口成为唯一入口
```

适用：替换组件实现、切换底层存储

### 两步修改

添加新代码和删除旧代码不在同一次提交。

```
提交 A：添加新代码（旧代码仍在）
提交 B：删除旧代码（新代码已验证）
```

适用：任何可能破坏功能的修改

### 小步提交

每步可独立验证，不积累大爆炸。

```
每步：修改 → 验证 → 提交
而不是：修改 → 修改 → 修改 → 验证 → 提交
```

## 技术债务分类

| 债务类型 | 特征 | 处理策略 |
|----------|------|----------|
| 故意欠债 | 为赶进度有意妥协 | 记录原因和还债计划 |
| 无意欠债 | 因认知不足产生 | 发现即修，不留过夜 |
| 过时债务 | 环境变化导致 | 评估影响，按优先级处理 |
| 架构债务 | 设计缺陷累积 | 制定渐进式迁移方案 |

还债优先级：影响用户的核心问题 > 阻碍开发效率的瓶颈 > 增加故障风险的隐患 > 降低可读性的混乱

## 代码气味检测

| 气味 | 检测信号 | 重构手法 |
|------|----------|----------|
| 长函数 | >50 行 | 提取子函数 |
| 深嵌套 | >3 层 | 卫语句提前返回 |
| 过多参数 | >4 个 | 引入参数对象 |
| 重复代码 | 三处以上相同逻辑 | 提取共享函数 |
| 霰弹式修改 | 一个变化点散布多处 | 收敛职责到同一模块 |
| 依恋情结 | 大量访问其他对象数据 | 移动行为到数据所在类 |

## 重构安全网

### 重构前

- 现有测试是否覆盖要重构的代码？
- 重构目标是否明确？
- 重构范围是否可控？

### 重构中

- 每步修改后测试是否仍然通过？
- 行为是否保持不变（重构不改行为）？
- 是否遵循两步修改原则？

### 重构后

- 新结构是否更清晰？
- 测试是否需要更新？
- 是否有遗留旧代码需要清理？
- 重构结果是否记录（commit message 说明原因）？
