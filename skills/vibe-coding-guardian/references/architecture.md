# 架构守护参考

加载条件：新建模块、添加依赖、修改系统边界、跨模块调用

---

## 模块边界规则

1. 每个模块有且只有一个变更理由（单一职责）
2. 模块间依赖方向：高层 → 低层，业务 → 基础设施
3. 跨模块调用必须通过公开接口，不直接访问内部实现
4. 模块内聚高、耦合低：改一个理由只动一个模块

边界检查：
- 新模块的职责能否用一句话描述？
- 依赖方向是否一致（无反向依赖、无循环依赖）？
- 公开接口是否最小化？
- 内部实现是否对外不可见？

---

## 依赖管理

引入新依赖前必须回答：

| 问题 | 判断标准 |
|------|----------|
| 是否必要？ | 能否用标准库或已有依赖解决？ |
| 维护状态？ | 最近 6 个月有无更新？ |
| 体积影响？ | 是否显著增加打包体积？ |
| 许可证？ | 是否与项目许可证兼容？ |
| 替代方案？ | 有无更轻量的替代？ |

原则：优先标准库 → 已有依赖 → 维护活跃的依赖。一个功能只用一个库。

## API 契约稳定性

| 变更类型 | 处理方式 |
|----------|----------|
| 新增可选字段 | ✅ 允许，带默认值 |
| 新增端点 | ✅ 允许 |
| 修改字段语义 | ⚠️ 需版本升级 |
| 删除字段 / 修改返回类型 | ❌ 禁止（除非主版本升级） |
| 修改错误码 | ⚠️ 需评估影响范围 |

契约测试 > 实现测试。如果只能写一种测试，写契约测试。

---

## 数据流原则

1. 数据流向清晰可追踪，不存在隐式数据通道
2. 避免双向依赖和循环引用
3. 状态变更必须可观测（事件、日志、状态机）
4. 事件/消息必须有明确的 schema
5. 数据所有权清晰：每个数据有且只有一个写入者

---

## 分层架构守则

```
表现层（UI/Controller）
    ↓ 只调用
应用层（Use Case/Service）
    ↓ 只调用
领域层（Domain/Entity）
    ↓ 只调用
基础设施层（Repository/External）
```

- 表现层不直接访问基础设施层
- 领域层不依赖任何外层
- 基础设施层实现领域层定义的接口（依赖倒置）
- 跨层调用只通过接口，不通过实现
