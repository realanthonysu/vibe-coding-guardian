# Context Engineering 参考

加载条件：任务涉及复杂代码库、多模块改动、或 AI 输出不符合业务约束时

---

## 什么是 Context Engineering

> Context Engineering = 在提问之前，把模型可能需要的一切都准备好。

这不是要加更多文字，而是要加**正确的文字**。

### 与 Prompt Engineering 的区别

| 维度 | Prompt Engineering | Context Engineering |
|------|-------------------|---------------------|
| 解决的问题 | "怎么说"（How to ask） | "给什么"（What to know） |
| 核心动作 | 优化措辞、添加示例、设定角色 | 收集项目信息、整理上下文、准备约束 |
| 失败原因 | 措辞模糊、缺少示例 | AI 缺乏项目背景、不了解业务约束 |
| 适用场景 | 单次问答、代码片段生成 | 复杂任务、多文件改动、功能实现 |
| 效果 | 影响单次输出质量 | 决定 AI 是否具备正确执行的前提条件 |
| 类比 | 写好一封信 | 确保收信人了解背景信息 |

**模型能力是天花板，上下文质量决定你离天花板有多远。**

## 上下文收集框架

### 1. 代码库探索

让 AI 先探索代码库，再执行任务：

```
请探索这个代码库，帮我理解：
1. 项目结构和技术栈
2. 与 [任务描述] 相关的文件和模块
3. 现有类似功能的实现方式
4. 代码风格和架构模式
5. 关键约束（接口、类型、配置）

不要修改任何代码，只输出你的理解。
```

### 2. 上下文整理

收集到信息后，整理出执行任务所需的上下文：

| 上下文类型 | 具体内容 | 示例 |
|-----------|---------|------|
| 文件路径 | 需要修改的文件 + 依赖的文件 | `src/auth/login.ts`, `src/types/user.ts` |
| 接口契约 | 函数签名、API schema、数据类型 | `POST /api/login → { token, user }` |
| 业务规则 | 该功能的业务逻辑约束 | "密码必须 8 位以上，包含大小写" |
| 代码风格 | 命名、格式、架构模式 | "使用 PascalCase 命名组件，函数式组件" |
| 技术约束 | 版本、性能、安全考量 | "Node 18+, 响应时间 < 200ms" |
| 类似实现 | 项目中已有的相似功能 | "参考 `src/auth/register.ts` 的错误处理模式" |

### 3. 上下文注入

将整理好的上下文 + 任务描述一次性输入：

```
## 项目上下文
- 技术栈: React + TypeScript + Tailwind
- 相关文件: src/auth/login.ts, src/types/user.ts
- 现有模式: 参考 src/auth/register.ts 的错误处理

## 接口契约
POST /api/login
Request: { email: string, password: string }
Response: { token: string, user: User } | { error: string }

## 业务规则
- 密码必须 8 位以上，包含大小写字母和数字
- 登录失败返回 401，不区分"用户不存在"和"密码错误"

## 任务
实现登录功能，保持与现有代码风格一致。
不要修改 register.ts 或其他无关文件。
```

## 上下文质量检查清单

在执行任务前，确认以下问题：

- [ ] **相关性**: 提供的上下文是否与任务直接相关？（排除无关信息）
- [ ] **完整性**: AI 是否具备执行任务所需的全部信息？
- [ ] **准确性**: 上下文是否反映当前代码库的真实状态？
- [ ] **简洁性**: 是否去掉了冗余信息，保持精简？
- [ ] **约束明确**: 业务规则、技术约束是否清晰列出？

**缺失任何一项 → 先补充，再执行。**

## 常见上下文陷阱

| 陷阱 | 表现 | 解药 |
|------|------|------|
| 上下文断裂 | AI 不知道昨天做了什么决策 | 在任务开始时重新注入关键上下文 |
| 信息过载 | 给了太多无关信息，AI 迷失重点 | 只给与任务直接相关的上下文 |
| 假设 implicit | 以为 AI "应该知道"某些背景 | 显性化所有假设，不要默认 AI 了解 |
| 上下文过期 | 代码库已变，但给的上下文是旧的 | 每次任务前重新探索代码库 |
| 跨工具隔离 | 在不同 AI 工具间切换，上下文丢失 | 用文档/笔记沉淀关键上下文 |

## 上下文管理最佳实践

### 小步迭代，持续验证

反对"一键生成整个应用"。正确流程：

```
1. 给 AI 一个简单、明确的任务
2. AI 生成代码
3. 快速验证（lint + type check + 手动测试）
4. 发现问题 → 精准定位 → 给 AI 明确的修正指令
5. 完成 → 下一个任务
```

### 精准修正模板

当验证发现问题时，不要完整粘贴错误信息说"帮我修"。正确做法：

```
## 问题定位
文件: user_service.py, 第 47 行
错误: AttributeError: 'NoneType' object has no attribute 'id'

## 我的分析
user_service.get_by_email() 可能返回 None，
但调用方没有检查。

## 修正指令
请在第 45-50 行添加 None 检查：
- 如果用户不存在，抛出 UserNotFoundError
- 不要修改其他代码
```

### 上下文沉淀

对于重要决策和关键上下文，沉淀到文档中：

- 架构决策记录（ADR）
- 模块边界说明
- 接口契约文档
- 业务规则清单

这样可以在后续任务中快速复用，避免每次从零开始。

## 多层上下文管理

对于复杂项目，上下文应分层管理：

```
项目层（全局上下文）
  ├── 技术栈、架构模式、代码规范
  └── 模块划分、依赖关系
  
模块层（局部上下文）
  ├── 模块职责、边界、接口
  └── 模块内关键约束和业务规则
  
任务层（单次上下文）
  ├── 本次任务相关文件
  └── 本次任务的特定约束
```

**执行规则**:
- 项目层 → 长期沉淀，每次任务引用
- 模块层 → 模块改动时更新
- 任务层 → 每次任务前收集，任务后可丢弃
